# Requirement 2 — Exception Handling 方案说明

## 目标

审计要求系统在遇到错误输入时，必须能够**安全处理异常而不终止程序**。

重点场景包括：

- 负数金额
- 空白姓名
- 非法帐号 / 非法客户编号
- 非法付款金额

本次目标不是重写系统，而是在现有结构上做**最小改动**，补齐异常处理边界，使系统更稳健并满足后续 Code Review 与 Unit Testing 要求。

---

## 一、我对 Requirement 2 的理解

这条需求的核心不是“简单加几个 `try-catch`”，而是：

> 让系统在出现非法输入或业务异常时，能够给出明确错误提示，并继续运行，而不是直接中断。

也就是说：

- 非法输入要被识别
- 非法状态要被阻止
- 错误要被正确捕获
- 主流程不能因为异常而退出
- 用户要得到可理解的提示信息

---

## 二、当前系统已经具备的异常处理能力

根据当前代码，系统已经具备一部分基础异常处理能力。

### 1. `Main.java` 中已有的处理

当前输入方法已经能避免很多控制台输入导致的程序崩溃：

- `readIntInRange()` 捕获 `NumberFormatException`
- `readPositiveInt()` 捕获 `NumberFormatException`
- `readPositiveAmount()` 捕获 `NumberFormatException`
- `readNonNegativeAmount()` 捕获 `NumberFormatException`

因此，用户输入非数字时，系统通常不会直接崩溃，而是提示后重新输入。

同时，主流程中也已经存在部分业务异常捕获：

- 创建贷款时，对 `new Loan(...)` 使用了 `try-catch`
- 还款时，对 `makePayment(...)` 使用了 `try-catch`

### 2. `Loan.java` / `DefaultLoanRules.java` 中已有的处理

当前领域层采用的是 **fail-fast** 方式：

- 数据非法时立即抛出异常
- 不允许对象进入非法状态

目前已经能处理的异常场景包括：

- 姓名为空或格式非法
- 贷款金额小于等于 `0`
- 初始已付款金额小于 `0`
- 还款金额小于等于 `0`
- 已完成贷款继续还款

这说明系统已经有“异常抛出”能力，但“异常分类”和“统一处理边界”还不够完整。

---

## 三、当前仍然存在的问题

虽然系统已经比最初稳定，但从审计角度看，仍有以下问题：

### 1. 异常类型过于单一

目前大多数错误都使用 `IllegalArgumentException`。

这会带来两个问题：

- 调用方只能知道“参数非法”，但不容易区分到底是：
  - 姓名错误
  - 金额错误
  - ID 错误
  - 已完成贷款错误
- 后续写单元测试时，错误分类不够清晰

### 2. 异常处理边界还不够统一

当前只有部分业务流程做了 `try-catch`。

如果未来新增菜单逻辑、别的 UI 层、或者直接从测试/其他入口调用业务类，而没有显式捕获异常，程序仍有可能终止。

### 3. 非法 ID 还不是“统一异常机制”的一部分

当前“非法帐号 / 非法客户编号”的处理主要是在 `Main.java` 里通过：

- `findLoanIndexById(...)`
- `index == -1`
- 然后打印 `Invalid customer ID.`

它目前是“流程判断”，不是“统一异常语义”。

### 4. 缺少总兜底异常处理

当前系统主要捕获的是预期异常。

如果未来出现某个未预料到的运行时异常，主菜单流程仍然有可能直接退出。

---

## 四、Requirement 2 的最小改动方案

### 总体思路

在不重写系统的前提下，沿用当前结构：

- `Main` 继续负责输入与错误提示
- `Loan` / `interfaces.LoanRules` 继续负责业务校验
- 通过补充“异常分类 + 统一捕获边界”，让系统更稳健

### 方案原则

- 不删除原有功能
- 不重构菜单结构
- 不改变当前主要业务流程
- 不把显示逻辑塞回实体类
- 尽量复用现有 `interfaces.LoanRules` 设计
- 只增强异常可读性和处理一致性

---

## 五、建议的实现方向

### 1. 保留领域层“抛异常”的职责

`Loan` 与 `DefaultLoanRules` 仍然负责：

- 判断输入是否合法
- 检测业务状态是否合法
- 在非法时抛出异常

这部分不建议反过来改成“吞异常”，因为那会让领域层职责变混乱。

也就是说：

- 业务层负责发现错误
- `Main` 负责向用户展示错误

这样的分层是合理的。

### 2. 在 `Main` 中明确统一异常处理边界

建议在现有基础上继续保留：

- 创建贷款时的 `try-catch`
- 还款时的 `try-catch`

并额外增加一层**主流程兜底异常处理**，确保：

- 即使未来有未预期异常出现
- 程序也不会直接崩溃退出
- 至少能给出通用错误提示，并继续或安全结束当前轮次

### 3. 对常见错误引入更清晰的异常分类

这是 Requirement 2 最值得补强的部分。

建议最小新增若干自定义异常，例如：

- `exceptions.InvalidCustomerNameException`
- `exceptions.InvalidAmountException`
- `exceptions.InvalidCustomerIdException`
- `exceptions.LoanAlreadyCompletedException`

这样做的好处：

- 更容易区分错误原因
- 更方便在 `Main` 中针对性提示
- 更方便后续单元测试断言
- 更容易通过审计与 Code Review

### 4. 把“非法帐号 / 非法客户编号”纳入统一处理思路

当前项目里没有独立的 Account 类，因此这里更准确地说是：

- 非法 `Customer ID`
- 非法 `Loan ID`

后续实现时，建议不要只保留 `index == -1` 这种分散判断，应该逐步统一为：

- 查询失败 -> 抛出统一异常
- 调用层捕获 -> 输出明确提示

这样能让“ID 错误”与“金额错误”“姓名错误”一样，成为正式的异常处理场景。

### 5. 统一错误消息文本

建议把错误提示尽量统一，例如：

- 姓名错误 -> 固定、清晰的提示
- 金额错误 -> 固定、清晰的提示
- ID 错误 -> 固定、清晰的提示
- 已完成贷款 -> 固定、清晰的提示

这样做对后续测试很重要，因为：

- 行为更一致
- 测试断言更稳定
- 用户体验更统一

## 六、建议的修改范围（仅方案，不是代码实现）

### 预计会涉及的文件

#### `src/Main.java`

主要职责：

- 捕获并处理异常
- 向用户显示错误提示
- 作为异常处理的主边界

预计调整点：

- 保留现有输入循环
- 增强菜单主流程的总兜底异常处理
- 对不同异常类型输出更明确的信息

#### `src/Loan.java`

主要职责：

- 保持实体合法状态
- 在检测到非法业务行为时继续抛出异常

预计调整点：

- 尽量少改
- 如有必要，改为抛出更明确的自定义异常类型

#### `src/DefaultLoanRules.java`

主要职责：

- 集中执行业务校验规则
- 抛出更有语义的异常

预计调整点：

- 把原本的 `IllegalArgumentException` 按场景拆分
- 保持当前业务规则不变

#### 新增异常类文件

可能新增：

- `src/exceptions.InvalidCustomerNameException.java`
- `src/exceptions.InvalidAmountException.java`
- `src/exceptions.InvalidCustomerIdException.java`
- `src/exceptions.LoanAlreadyCompletedException.java`

是否全部新增，取决于最终要控制的改动规模。

如果要最小化修改，也可以先只引入最关键的 2~3 个异常类。

## 七、推荐的实施顺序

### Step 1

梳理当前所有非法输入与非法业务状态，明确分类：

- 输入解析错误
- 姓名错误
- 金额错误
- ID 错误
- 已完成贷款错误

### Step 2

为最常见错误引入自定义异常类型。

### Step 3

让 `interfaces.LoanRules` / `Loan` 抛出更明确的异常，而不是全部混用 `IllegalArgumentException`。

### Step 4

在 `Main` 中统一捕获这些异常，并给出可读提示。

### Step 5

在主流程外层增加一个轻量级兜底异常处理。

### Step 6

回归测试所有错误输入，确认：

- 程序不崩溃
- 菜单能继续运行
- 错误提示正确
- 原有正常功能不受影响

---

## 八、验收标准

完成 Requirement 2 后，至少应满足：

### 1. 功能层

以下场景不会导致程序终止：

- 输入负数金额
- 输入空白姓名
- 输入非法客户编号
- 输入非法还款金额

### 2. 结构层

- 业务层继续负责抛异常
- 表现层负责处理和展示异常
- 异常类型具备一定语义化能力

### 3. 用户体验层

- 错误提示明确
- 系统不会突然退出
- 用户可以继续后续操作

### 4. 质量层

- 能通过 Code Review
- 能为 Requirement 3 的 Unit Testing 提供明确测试点

---

## 九、一句话总结

`Requirement 2` 的最小改动重点不是推翻现有校验，而是：

> 在当前 `Loan + interfaces.LoanRules + Main` 结构上，补齐异常分类、统一捕获边界和清晰错误提示，使系统在遇到非法输入时能够安全运行而不崩溃。
